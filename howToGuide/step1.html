Getting Started with Android Testing

Why should we test our Android applications?

As of February 18, 2016, there are 1,999,282 Android applications available for download on Google Play. Formerly known as the Android Market, the Google Playstore is the central hub for applications for the Android operating system. There were 50 billion downloads as of July 2013. Currently, the number of applications in the Google Playstore exceeds that in the Apple App Store. Since August 2015, there has been consistently over 40,000 Android applications published per month. [1][2][3]

The above is clearly an indication of the sheer volume of Android application consumption in the general public, and should serve as motivation to test our Android applications thoroughly to ensure positive user experiences. To quote the computer scientist Edsger Dijkstra: [4]

Testing shows the presence, not the absence of bugs
Edsger Wybe Dijkstra

Although we can't eliminate all bugs, testing can increase our confidence that our applications are performing correctly.

Unit Testing

Unit testing is checking logical components (methods, classes etc) of an application in isolation. Testing individual components thoroughly can help ensure these parts work correctly before considering their interaction with other components.

In Android, you can perform unit tests locally, or as instrumented unit tests. The following describes the difference between these two types of unit tests:

Local unit tests
These unit tests are independent of the user interface components, and thus do not invoke the Android AVD (virtual device/emulator) or physical device. This means that these tests run quickly.
Instrumented unit tests
Unlike local unit tests, instrumented unit tests need to invoke the an Android device because they test the Android UI. When you run these tests, you will be prompted to open either an emulator or physical device. [5]
